Routing and navigation

Roselt.js supports query routing by default and pathname routing when you need it.

Query routing is the practical default because it behaves well on static hosts. Pathname routing is available when your deployment can rewrite arbitrary URLs back to the main entry document.

Default query routing

With the default configuration, Roselt.js reads the page search parameter.

?page=home
?page=about
?page=docs/getting-started

If the parameter is missing, Roselt.js uses the navigation page root's page attribute or home. The default start path requires a routed page root marked with navigate.

Switch to pathname routing

Set routingMode: "pathname" when you want clean paths and your host can serve the entry page for every route request.

window.Roselt.start({
  routingMode: "pathname"
});

In pathname mode, a page named docs/getting-started resolves to /docs/getting-started.

Use explicit routes when filenames are not enough

Automatic routes are the default. Pass a routes array only when you need named routes, custom file locations, route metadata, or pathname parameters.

window.Roselt.start({
  routingMode: "pathname",
  routes: [
    {
      name: "blog-post",
      path: "/blog/:slug",
      html: "pages/blog/post.html",
      module: "pages/blog/post.js"
    }
  ]
});

When a route path contains parameters, Roselt.Page.params exposes them inside the mounted page module.

Navigation helpers

Real links

Any standard anchor with an internal URL keeps working. Roselt.js intercepts same-origin navigation and renders the next page in place.

href on other elements

Buttons or other elements can navigate too when they carry an href attribute, as long as they are not downloads or external targets.

navigate() and href()

The Roselt.js runtime exposes helpers for programmatic navigation and URL construction globally and on Roselt.Page while a route is mounted. Prefer Roselt.Page.href() when you need a generated URL in page code.

404 behavior

If Roselt.js cannot load the requested page HTML, it falls back to a built-in not-found screen or to your custom pages/404.html page when that file exists.

Deep link reality

Query routing avoids the classic static-host problem where /docs/getting-started 404s on refresh. If you need GitHub Pages compatibility, query routing is the safer choice.
