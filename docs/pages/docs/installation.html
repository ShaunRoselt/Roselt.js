Installation

Choose between a local package workflow and a zero-config script include.

Roselt.js can be consumed either as a package you build into your own app or as the prebuilt global bundle. The right choice depends on how much control you want over your surrounding toolchain.

Option 1: scaffold a new app

npm create roselt-js@latest my-app
cd my-app
npm install
npm run dev

This is the recommended starting point if you want the official starter shell, route structure, and development script without setting them up by hand.

Option 2: install from npm

npm install roselt-js

Use the package directly when Roselt.js is one part of an existing toolchain or repository.

Option 3: use the global bundle

The quickest no-build path is a script tag that exposes window.Roselt directly.

<script src="https://cdn.jsdelivr.net/gh/ShaunRoselt/Roselt.js@main/dist/roselt.js"></script>
<script>
  window.Roselt.start();
</script>

Repository workflow

If you are working inside this repository, install dependencies and refresh the bundles with the existing scripts:

npm install
npm run build
npm run serve

What the local scripts do

Script Purpose
npm run build Builds dist/roselt.js and dist/roselt.min.js with esbuild targeting ES2022.
npm run check Runs syntax checks for the build and local server scripts.
npm run serve Starts the simple local server so you can preview docs/ and other repo files without adding another dev dependency.

When to keep the package unbundled

Latest-browser assumption

Roselt.js is built for modern browsers and ships ES2022 output. If your project needs legacy support, that should be solved in your build pipeline rather than inside the framework itself.

One browser bundle

The browser-facing story is intentionally narrow: load dist/roselt.js or dist/roselt.min.js, then call window.Roselt.start().
