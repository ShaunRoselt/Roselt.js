Getting started

Get a Roselt.js app running with one entry file and a pages/ folder.

Roselt.js is optimized for the simplest working shape: an index.html entrypoint, a roselt page root, and routed page files. If you keep query routing, the same setup works cleanly on GitHub Pages.

Fastest start: scaffold a project

The quickest way to get a working Roselt.js app is the official scaffold command.

npm create roselt-js@latest my-app
cd my-app
npm install
npm run dev

The generated app includes an index.html entrypoint, a starter shell, a home route, a 404 route, and a dev script powered by roselt serve.

Start from the minimum

If you want to understand the framework from first principles, create an index.html at the root of your app and use the default directories directly.

<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>My Roselt.js App</title>
  </head>
  <body>
    <roselt page="home" navigate></roselt>

    <script src="https://cdn.jsdelivr.net/gh/ShaunRoselt/Roselt.js@main/dist/roselt.js"></script>
    <script>
      window.Roselt.start();
    </script>
  </body>
</html>

The routed page root must include navigate. If you later add more than one roselt page element, Roselt.js will still only use the one marked with navigate for normal navigation.

Add your first route

Roselt.js resolves page files from the folder layout. A default home route maps to pages/home.html and optionally pages/home.js and pages/home.css.

pages/
  home.html
  home.js
  home.css

A minimal pages/home.html file can be pure HTML.

<section>
  <h1>Home</h1>
  <p>Roselt.js rendered this from pages/home.html.</p>
  <a href="?page=docs/getting-started">Read the docs</a>
</section>
Browser global

When you load the browser bundle from a CDN or a local file, Roselt.js now exposes window.Roselt.start() directly instead of nesting the API under another namespace.

Understand the startup behavior

Resolve the shell

Roselt.js resolves any roselt section="..." includes around the page root once at startup so they stay mounted during navigation.

Match a route

With query routing, Roselt.js reads the page search parameter and maps it directly to a page file path under pages/.

Load behavior and styles

Roselt.js loads the page module if it exists, automatically tries the sibling CSS file, and then swaps the page markup into the page root.

Recommended next steps

  • Use the scaffolded app when you want the fastest path, and the manual setup when you want to see every moving part.
  • Add a shared header and footer with roselt section="site-header" and roselt section="site-footer".
  • Set Roselt.Page.Title and Roselt.Page.Description inside each page module once you add documentation or marketing pages.
  • Keep query routing if you want the simplest GitHub Pages deployment story.
Good default

Use ?page= URLs until you have a strong reason to switch. Roselt.js pathname routing is useful, but static hosting is easiest when every deep link can stay inside a single index.html request.
