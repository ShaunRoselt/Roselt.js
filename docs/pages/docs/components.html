Components

Roselt.js resolves custom elements lazily from components/.

Components are standard custom elements. Roselt.js discovers the tags present in the rendered DOM and loads the script that registers them when needed.

Automatic component discovery

A tag such as <ui-button> maps to components/ui-button.js by default when Roselt.js sees it in the shell or rendered page. The script should register the element itself.

Roselt.defineComponent("ui-button", class UiButton extends HTMLElement {
  connectedCallback() {
    this.innerHTML = `<button type="button">${this.getAttribute("label") || "Button"}</button>`;
  }
});

Supported component definition shapes

Shape How Roselt.js handles it
class MyElement extends HTMLElement Registers the constructor immediately if the custom element name is still available.
() => MyElement Waits until the tag appears, then resolves the loader and uses the returned constructor.
"my-element.js" Loads and executes the script file, which should call Roselt.defineComponent(...) or customElements.define(...).

Manual registration helpers

Export Use it for
defineComponent(tagName, constructor) Register a known custom element constructor ahead of time.
lazyComponent(tagName, loader) Register a lazy loader that returns either a constructor or a classic script path when Roselt.js encounters the tag.
globalComponentRegistry Access the shared registry directly when you need lower-level control.

Page-specific component declarations

The Roselt.Page.Components setter lets a page register component definitions before that route's components are resolved. URL strings are resolved relative to the page module file.

Roselt.Page.Components = {
  "ui-banner": "../components/ui-banner.js"
};
Keep the platform model

Roselt.js does not invent its own component authoring system. If you already understand custom elements, you understand how the framework expects components to behave.
