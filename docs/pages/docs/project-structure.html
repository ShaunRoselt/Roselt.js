Project structure

Roselt.js reads your directory layout as application structure.

The framework is intentionally opinionated here. Instead of maintaining a separate route registry for common cases, you usually let Roselt.js resolve requested pages, sections, and components from three default directories.

Default layout

index.html
pages/
  home.html
  home.js
  docs/getting-started.html
sections/
  site-header.html
  site-header.js
  site-header.css
  site-footer.html
components/
  ui-button.js

Directory roles

Directory What belongs there
pages/ Routeable HTML files, optional page modules, and optional same-name CSS files.
sections/ Portable HTML fragments that can be resolved anywhere with roselt section="...", with optional same-name CSS and JS sidecars.
components/ Custom element modules loaded lazily when their tag names appear in the rendered DOM.

Nested pages are just nested folders

If the URL requests ?page=blog/post, Roselt.js looks for pages/blog/post.html and pages/blog/post.js.

No extra route object required

File-based page resolution is the default. You only need explicit routes when you want custom paths, pathname parameters, extra route metadata, or custom file locations.

Custom directories remain available

Roselt.start() accepts pagesDirectory, sectionsDirectory, and componentsDirectory when you want a different shape. The defaults are the fast path, not a hard limitation.

window.Roselt.start({
  pagesDirectory: "app/pages",
  sectionsDirectory: "app/sections",
  componentsDirectory: "app/components"
});
