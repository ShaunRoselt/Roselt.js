Deployment

Roselt.js is easiest to deploy where one index.html can own the experience.

Query routing is the default recommendation because it avoids server rewrite requirements. That makes GitHub Pages and similar static hosts a natural fit for Roselt.js apps and documentation sites.

Recommended GitHub Pages model

  • Keep the app entrypoint as index.html.
  • Use query routing links such as ?page=home and ?page=docs/getting-started.
  • Publish the built or authored site from the folder GitHub Pages serves.
  • Keep your asset, page, section, and component paths relative to the served app directory.

Preview locally before publishing

npm install
npm run build
npm run serve

Then verify the landing page, representative docs URLs, refresh behavior, and the custom 404 page.

Pathname routing deployment requirements

If you switch to routingMode: "pathname", your host must rewrite route requests back to the app entrypoint. Without rewrites, direct loads such as /docs/getting-started will return a server 404 before Roselt.js can start.

Base path note

When the app is served from a subdirectory, set basePath explicitly if document.baseURI does not match the public app root.

Deployment checklist

Verify the bundle

Refresh dist/ before publishing so the public app references the current framework code.

Test deep links

Load representative query URLs directly to confirm the host preserves the search string and serves the entry page.

Check metadata

Open the rendered head output and confirm titles, descriptions, and canonical links match the public routes you expect users to share.
